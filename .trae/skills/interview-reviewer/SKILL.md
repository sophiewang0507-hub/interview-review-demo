# Interview Reviewer

这个 Skill 用于把“面试题 + 候选人回答”转成结构化复盘结果，适用于求职辅导、模拟面试、表达优化、追问预测等场景。

## 适用时机

当用户出现以下意图时，应优先调用本 Skill：
- 想判断一段面试回答答得好不好
- 想知道问题在哪里、应该怎么改
- 想把回答改写成更像真实面试中的高质量表达
- 想预测面试官下一轮最可能追问的问题
- 想做多岗位、多题型的面试复盘或模拟面试训练

## 目标

基于输入的：
- `role`：目标岗位
- `question`：面试题
- `answer`：候选人原始回答

输出结构化结果：
- `score`：综合评分
- `level`：优秀 / 合格 / 需改进
- `summary`：总体评价
- `highlights`：答得好的点
- `problems`：主要问题与修正建议
- `betterAnswer`：更细化、更接近真实面试表达的改写示例
- `followUps`：下一轮最可能的追问
- `rubric`：多维评分
- `retrieval`：参考了哪些示例（RAG）

## 核心原则

### 1. 优先真实，不要编造
- 严禁凭空补充候选人未提供的关键事实，例如：
  - 公司名
  - 业务规模
  - 确定数值
  - 具体技术栈
  - 团队人数
- 信息不足时，必须保留【占位符】提示补充，例如：
  - `【指标】`
  - `【影响范围】`
  - `【技术方案】`

### 2. 输出必须像“真实面试里能说出口的话”
- 避免纯书面表达
- 避免空话，例如“我学习能力强”“我沟通能力好”
- 必须落在：
  - 我负责了什么
  - 我怎么做的
  - 为什么这么做
  - 怎么验证有效
  - 结果是什么

### 3. 按岗位和题型输出
- 不同岗位应该突出不同维度：
  - 后端：稳定性、性能、取舍、回滚、监控
  - 前端：性能、工程化、用户体验、可维护性
  - 产品：目标、指标、优先级、推进、复盘
  - 数据：口径、拆解、验证、可解释性
  - 算法/AI：实验设计、特征/模型、上线验证、漂移/回滚

### 4. 改写示例必须有结构
`betterAnswer` 默认至少包含：
- `【30 秒电梯版】`
- `【2 分钟标准版（STAR）】`
- `【我负责的部分】`
- `【可补充信息清单】`

## 推荐工作流

### 步骤 1：识别输入完整性
检查用户是否提供：
- 目标岗位
- 面试题
- 回答内容

若任一缺失，应提示补齐后再进入复盘。

### 步骤 2：识别岗位与题型
根据 `role + question` 判断：
- 岗位方向
- 题型类别，例如：
  - 行为题
  - 项目题
  - 故障处理
  - 性能优化
  - 系统设计
  - 指标分析
  - 增长实验
  - Prompt / RAG / 模型训练

### 步骤 3：调用示例库 RAG
优先参考当前工作区里的示例库：
- `server/knowledge/examples_seed.json`
- `server/rag.ts`

要求：
- 检索最相近的示例模板
- 只把示例作为“表达结构和颗粒度参考”
- 不能把示例中的具体事实直接套到用户身上

### 步骤 4：输出结构化复盘
结果需包含以下字段：

```json
{
  "score": 0,
  "level": "优秀",
  "summary": "",
  "highlights": [],
  "problems": [
    {
      "title": "",
      "detail": "",
      "fix": ""
    }
  ],
  "betterAnswer": "",
  "followUps": [],
  "rubric": [
    {
      "dimension": "",
      "score": 0,
      "comment": ""
    }
  ],
  "retrieval": []
}
```

### 步骤 5：优先提升 `betterAnswer`
如果用户重点想“优化输出质量”，优先加强：
- 更真实的口语表达
- 更强的可验证性
- 更清晰的 STAR 结构
- 更明确的“我负责的部分”
- 更具体的【占位符】提示补齐

## RAG 参考规范

### 当前项目中的知识来源
- `server/knowledge/examples_seed.json`
  - 存放按“岗位 × 题型”组织的 seed 示例
- `server/index.ts`
  - 运行时扩展为 60 条（校招/社招/专家）
- `server/rag.ts`
  - 负责关键词打分、TopK 检索与上下文拼接

### 使用原则
- RAG 是“参考模板”，不是事实来源
- RAG 应提升：
  - 输出稳定性
  - 表达颗粒度
  - 题型贴合度
  - 岗位匹配度
- RAG 不得导致：
  - 编造用户没说过的事实
  - 混入其他示例的具体指标
  - 强行套模板而忽视原始回答

## 输出风格要求

- 中文输出
- 简洁、专业、像真实资深 HR/面试官
- 不使用空洞模板化措辞
- 建议优先“先结论，再解释”

## 不适用场景

以下情况不建议直接调用本 Skill：
- 用户只是问某个概念的定义，不需要复盘
- 用户仅想做纯技术问答，不需要候选人表达优化
- 用户要做正式 ATS 简历解析而非面试回答复盘

## 示例触发语句

- “帮我看看这段面试回答答得怎么样”
- “你作为面试官，帮我复盘这段回答”
- “这道题我回答得不好，帮我改写一下”
- “帮我预测下一轮面试官会追问什么”
- “给我做一个面试复盘助手”

## 示例结果风格

### 好的输出应该像这样
- 先指出这段回答最大的短板
- 再说明为什么这个短板会影响面试官判断
- 再给出可直接复述的改写版本
- 最后给出尖锐但合理的追问

### 不好的输出
- 只给笼统建议，如“更自信一点”
- 只夸不指出问题
- 编造候选人没说过的数据
- 改写后仍然无法直接口述
